# 梅奥 MEIAO 当前版本 · 技术体检报告

> 日期:2026-06-13 · 性质:只读全栈体检(不改代码)
> 目标:为"先做稳"的升级路线提供证据基础。结论以读码 + 命令验证为准,推测部分已明确标注。

## 0. 一句话结论

**不是推倒重写的局面,是"病根集中、可分阶段治"的局面。**
技术栈很新(React 19 / Vite 7 / TS 5.9 / Temporal 1.16)无需更换;不稳定来源集中在约 5-6 个文件上,占全部 9 万行代码的极小比例。唯一确认的结构性大问题是**单行 JSON 数据模型(#4)**,中期值得做一次"把项目数据搬出单行"的改动(不是重写,是搬家)。

## 1. 通关标准(已与业主确认,作为"做稳"的验收线)

1. 看板上这几类报错降到接近 0:旧资源 404(已修)、`Pool is closed`(曾 17 次)、上游 provider 错误指纹(曾 21+)、stale 残留任务
2. PM2 重启从 203 次量级降到个位 / 零
3. "卡处理中 / 已完成被覆盖 / 重试中显示成失败"类用户反馈消失(#1/#3 已修,待线上验证)
4. 前端不再出现脏数据、乱序展示
5. 同一个 bug 不再复发——关键路径都有测试兜底(复发率→0)

## 2. 体检发现(七块)

### 体检1 · 技术栈与依赖 ✅ 已自验
- **技术栈很新,无需更换**:React 19、Vite 7、TypeScript 5.9、Temporal 1.16、mysql2、Radix UI、Zod 4。
- **安全**:4 个 high 漏洞,全部来自同一条**开发/打包工具链**(esbuild → vite → 两个开发插件),非运行时/数据暴露。中优先,排进路线,不抢救。
- **便宜高价值**:`npm run lint` 实际只跑 `tsc -b`(类型检查),**没开 ESLint**;而 ESLint 装了且带 react-hooks 插件(专抓前端状态/副作用 bug)。等于免费体检仪没插电。

### 体检2 · 体量与 churn ✅ 已自验
- 源码约 **89,785 行**,测试约 24,359 行。
- **两个巨型文件**:`server/index.mjs` 10863 行、`src/ShellMigratedApp.tsx` 7134 行。
- churn(改动最频繁)高度集中:index.mjs(49)、ShellMigratedApp.tsx(36)、providerGateway.mjs(30)、shellDataAdapter.ts(25)、appStateMerge.mjs(22)。
- **174 次提交里 124 次(71%)是修 bug**(健康项目约 20-40%)——"被 bug 拖累"的数字版。
- 结论:**不稳定不是摊在 9 万行里,而是挤在这少数几个文件上,且正好是已诊断病根所在地。**

### 体检3 · 剩余病根 #2/#4 ✅ 已自验
- **#4 单行 JSON 数据模型(崩溃总根源)**:`compactOneClickProjectForStorage`(appStateMerge.mjs:79-93)只剥 5 个字段,`results` 数组(含 imageUrl)原样保留;整棵项目树塞进 `app_states` 一行。写入点 `saveDbAppState`(index.mjs ~2645)用 `INSERT ... ON DUPLICATE KEY UPDATE` 全量序列化。
- **#2 正则猜状态**:`INVALID_ONE_CLICK_PLAN_PATTERNS`(oneClickPlanValidation.ts:15-31)共 **15 条**硬编码正则;**2 个定义点**(前端 oneClickPlanValidation.ts + 后端 appStateMerge.mjs:20),改一份漏一份。
  - 注:初步侦察曾报"3 份拷贝",自验为 2 份——之前重构已收敛过一次。

### 体检4 · 崩溃链路 ✅ 机制已验证(具体字节数为推测)
- 链路:项目越用越多 → 单行 JSON 越来越大 → 超过数据库单条写入上限 → `Pool is closed`(看板 17 次)→ 任务卡住 → 定时 reconcile 每 60s 重试又失败 → 死循环 → PM2 反复重启(看板 203 次)。
- 关键位置:`appStateMerge.mjs:66-141`(compact)、`index.mjs:2645-2649`(写入)、`index.mjs:2122-2132`(连接池)、`jobRuntime.mjs:170-172`(Pool is closed 识别)、`index.mjs:5308-5346`(stale job reconcile,60s 一次)。
- **边界声明**:崩溃**机制确凿**;"单行多大才爆、是否已在爆"需测量线上真实数据,只读阶段无法获取。**关键未验证项:`result.imageUrl` 到底是内联 base64 还是远程 URL**——这决定了瘦身策略,需在第 1 阶段优先测量。

### 体检5 · 脏数据来源 ✅ 关键两处已自验
- **prompt 缺失 → fallback 成"模块标签 / 项目名+序号"**:shellDataAdapter.ts:695、`getPrompt` 多层 `||` 链最终落到 fallbackTitle。
- **model 缺失 → 直接写死字符串 "旧任务"**:shellDataAdapter.ts:**696、926**(已亲眼确认两处)。
- 这些"占位值"被当真实历史持久化进库 → 即用户看到的脏数据 / 乱七八糟展示。
- 修复方向:fallback 值打"占位"标记,合并/展示/持久化时不当真值。

### 体检6 · 前端展示乱与交互卡 ✅ 关键结论已自验
- **两个巨型组件未做重渲染优化(无 memo)**:`ProjectCard`(2287 行)、`BottomInputBar`(2461 行)——父组件一变即全量重画 = 卡顿直接原因。
- **乱序**:主壳 `filteredProjects`(ShellMigratedApp.tsx:6812)在传给下游前不排序、`createdAt` 格式不统一(shellScopeFilters.ts 排序键可能失效);下游 ProjectListView 有排序兜底,故为"可能乱"而非"必乱"(已修正初步侦察的夸大)。
- 修复方向:两个大组件加 memo;排序键与时间格式统一在数据层。

### 体检7 · 测试盲区 ✅ 已自验
- **最大稳定风险(单行 JSON 写爆)既无代码保护、也无测试兜底**:无"超大就裁剪"的测试,因为代码里没有该保护。
- `Pool is closed` 只测了"识别"(jobRuntime.test.mjs:412),**未测"恢复"**。
- 两个万行文件几乎无法直接单测,只能靠 acceptance 间接覆盖。
- 结论:关键路径无安全网 = "同一个 bug 反复犯"的结构原因。

## 3. 治理清单(按 对稳定的伤害 × 修复成本/风险 排序)

| 优先级 | 治什么 | 对"稳"的伤害 | 成本/风险 |
|---|---|---|---|
| 🔴 P0 | #4 崩溃链路**止血**(写入大小守卫+裁剪历史、连接池恢复容错、reconcile 退避) | 最高 | 低-中 |
| 🔴 P0 | 脏数据 fallback **止血**(占位值不入库) | 高 | 低 |

| 🟡 P1 | #2 正则 → 结构化判定 + 两份合一 | 中 | 中 |

| 🟢 P2 | #4 数据模型规范化(治本,中期)+ 安全升级 + 巨型文件拆小 | 高(治本) | 高 |

## 4. 分阶段路线(每阶段独立见效、独立分支、独立验收)

- **第 1 阶段 · 止血周(约 1-2 周)**:#4 崩溃链路止血 + 脏数据 fallback 止血,每条配测试(首次把崩溃路径纳入安全网)。开工前先测量真实状态大小(探针先行)。
- **第 2 阶段 · 体验顺滑(约 1 周)**:两大组件加 memo、排序/时间格式统一、lint 接上 ESLint。
- **第 3 阶段 · 治本(2-4 周,中期)**:#4 数据模型规范化(项目数据搬出单行 JSON、按需加载)、#2 结构化判定。放在止血见效、有测试网之后。
- **第 4 阶段 · 工程卫生(穿插)**:安全漏洞升级、巨型文件按模块拆小。

## 5. 验证方式(替代"线上空等观察")

- 每条修复走 TDD:先写失败测试、再实现、再验证(red→green)。
- 崩溃路径用真实数据探针测量后再定阈值,不靠猜。
- 对照云上日志诊断看板的报错指纹做前后对比,作为"通关标准"的实测依据。
