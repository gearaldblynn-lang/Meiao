# 视频诊断系统数据验证型 MVP 设计

## 目标

在现有短视频模块下新增一个并列子功能 `视频诊断系统`，先做一版方便测试的数据验证型 MVP。

这一期的核心不是做完整商业化页面，而是尽快验证三件事：

- 抖音和 TikTok 链接到底能稳定拿到哪些真实字段
- 这些真实字段是否足以支撑“像后台诊断一样详细”的报告
- 哪些结论能作为平台字段证据输出，哪些只能作为推断输出

## 产品定位

`视频诊断系统` 是一个二合一工作台：

- `字段勘探台`：负责拉取接口、展示原始 JSON、列出字段路径、提示关键字段缺失
- `诊断报告台`：基于勘探结果生成带证据链的诊断报告

任何诊断都必须建立在勘探结果之上，不能跳过勘探直接出结论。

## 本期范围

### 包含

- 在短视频模块中新增 `视频诊断系统` 子功能入口
- 支持输入 `TikTok` 或 `抖音` 视频链接
- 支持选择分析项目
- 后端调用 spider 网关的首批接口做字段勘探
- 保存原始响应、字段路径和统一聚合结果
- 前端展示勘探结果和基础诊断报告
- 报告明确区分 `真实字段证据` 与 `推断分析`

### 不包含

- App 设备抓包
- 自动生成设备签名
- 批量链接诊断
- 完整权限管理页面
- 历史报告库和横向对比分析

## 平台与数据范围

### TikTok

本期作为优先验证平台，优先接入以下 spider 接口：

- `tiktok/video-by-url-v2`
- `tiktok/video-v3`
- `tiktok/user-profile`
- `tiktok/user-posts-v3`
- `tiktok/video-comments`

原因：

- 已有分享链接直查能力
- 接口组合较适合“输入链接直接诊断”
- 更容易形成完整的勘探闭环

### 抖音

本期同步做基础接入，优先接入以下 spider 接口：

- `douyin/video-info`
- `douyin/user-info`
- `douyin/video-list`
- `douyin/play-count`

约束：

- spider 文档里暂未发现明确的“分享链接直出视频详情”接口
- 抖音链路可能需要额外的 URL 解析或视频 ID 提取能力

本期设计要求抖音链路允许“部分完成”：

- 能提取到 `aweme_id` 时继续勘探
- 无法解析时返回明确错误，不伪造结果

## 接入方式

本期 UI 只暴露两类接入方式：

- `Spider API`
- `网页登录态增强（预留）`

其中：

- `Spider API` 本期可实际跑通
- `网页登录态增强` 本期只预留配置位和架构扩展位，不要求完整打通

## 页面结构

在现有短视频模块下形成两个并列子功能：

- `短视频分镜配置`
- `视频诊断系统`

`视频诊断系统` 页面采用左右双栏：

### 左栏：输入与控制区

- 平台选择
- 视频链接输入
- 分析项目多选
- 接入方式选择
- `开始勘探` 按钮
- `生成诊断报告` 按钮

### 右栏：结果区

上半区为 `字段勘探台`：

- 已调用接口列表
- 每个接口状态
- 原始 JSON 预览
- 字段路径列表
- 缺失关键字段提示

下半区为 `诊断报告台`：

- 诊断总览
- 真实字段证据卡片
- 推断分析卡片
- 优化建议

## 用户流程

### 1. 输入阶段

用户选择平台、输入视频链接、勾选分析项目。

### 2. 勘探阶段

点击 `开始勘探` 后，后端按平台预设探测器顺序拉取数据。

勘探完成后必须产出三份结果：

- `raw responses`
- `flatten fields`
- `normalized aggregate`

### 3. 诊断阶段

只有勘探成功后才允许点击 `生成诊断报告`。

若关键字段缺失，允许生成降级报告，但必须显式标记：

- 哪些关键字段缺失
- 哪些判断基于真实字段
- 哪些判断属于推断

## 后端数据流

### Step 1：链接标准化

后端接收：

- `platform`
- `url`
- `analysisItems`
- `accessMode`

先做 URL 标准化，提取：

- `normalizedUrl`
- `videoId`
- `authorId`

若某平台无法解析出所需 ID，则中断并返回结构化错误。

### Step 2：平台勘探器执行

按平台执行一组探测器，每个探测器都要记录：

- 请求目标接口
- 请求参数
- 成功/失败状态
- 原始响应
- 提取出的关键字段

### Step 3：字段摊平

对每个原始响应执行 flatten，产出字段路径列表，供页面展示与后续调试。

### Step 4：统一聚合

把多接口结果合并成统一 JSON，供诊断引擎消费。

### Step 5：诊断生成

诊断引擎消费统一 JSON，输出：

- 证据摘要
- 风险/机会判断
- 建议动作

## 诊断输出规则

本期最重要的规则是“分层输出”。

### 真实字段证据

仅允许引用真实接口中存在的字段，必须能追溯到：

- 来源接口
- 字段路径
- 原始值

### 推断分析

当平台没有直接给出真值时，允许基于真实字段和内容信号做推断，但必须标注为推断，不能伪装成后台字段。

### 行动建议

建议必须尽量基于证据或推断给出，不写空泛套话。

## MVP 首批诊断项

本期诊断项控制在以下范围：

- 基础播放与互动表现
- 账号基础画像
- 评论样本反馈
- 近期作品对比信号
- 可见标签/音乐/文案等内容线索
- 基于证据的优化建议

本期不承诺以下结论一定具备平台真值支撑：

- 审核状态
- 限流状态
- 原创判定
- 风控等级

若这些字段真实存在则展示；若不存在则只能作为推断项或直接标记缺失。

## 测试便利性要求

这版实现必须优先服务测试，因此要求：

- 一个页面内即可完成输入、勘探、看结果
- 原始 JSON 必须可见
- 字段路径必须可见
- 接口失败必须清晰显示，不隐藏失败
- 报告中的关键结论必须能回看证据

## 实施顺序

### 第一阶段

- 补充状态结构与模块子模式
- 新增 `视频诊断系统` 最小页面
- 新增后端勘探入口

### 第二阶段

- 先打通 TikTok 勘探闭环
- 再接抖音基础链路

### 第三阶段

- 接入统一 JSON
- 生成第一版诊断报告

### 第四阶段

- 根据测试结果补缺失字段提示、错误态和可读性

## 风险与降级

### 风险 1：接口字段远少于预期

降级策略：

- 页面保留勘探结果输出
- 报告弱化“平台真因”表述
- 重点展示字段缺失情况

### 风险 2：抖音链接无法稳定解析

降级策略：

- 抖音先支持 `aweme_id` 驱动
- 页面明确提示“链接解析能力待补”

### 风险 3：登录态增强本期尚未接通

降级策略：

- UI 仅展示预留入口
- 实际流程仍以 Spider API 为主

## 验收标准

- 用户能在短视频模块切换到 `视频诊断系统`
- 输入 TikTok 链接后，能完成一次完整勘探并看到结果
- 输入抖音链接后，系统能明确进入可勘探或解析失败分支
- 页面能展示原始 JSON、字段路径和基础报告
- 报告中至少一部分结论带有真实字段证据
- 缺失字段会被显式提示，不会被系统伪装成已有结论
