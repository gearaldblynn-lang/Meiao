# 全局外壳压缩与智能体中心高度重分配设计

## 目标

把当前系统从“顶部承担模块识别 + 右上角堆叠全局入口 + 页面内部再叠多层头部”的结构，改成更轻、更稳定的工作台布局，直接释放内容区高度。

这次设计解决四个问题：

- 模块识别重复：左侧导航已经能表达当前模块，顶部再显示模块标题和副标题属于重复占位
- 全局入口分散：版本、服务状态、通知、帮助、账号入口散落在右上角，横向噪音高
- 内容区被多层头部挤压：尤其是智能体中心、工作室、聊天和日志页，可视区域不足
- 页面信息层级不统一：全局壳层、模块页头、子页面头部都在抢视觉层级

## 设计结论

采用“方案 B”：

- 顶部保留极薄的全局栏，但不再承担模块识别和主要功能入口展示
- 左侧导航继续承担当前模块识别
- 版本号、服务状态、通知、帮助、系统设置、账号管理、退出登录，全部收口到左侧导航底部用户区
- 智能体中心和其他模块内部页头进一步压薄，把高度优先让给工作区、聊天区、列表区

## 不变项

本次只调整外壳布局和信息架构，不改以下内容：

- 数据结构
- 服务端接口协议
- 权限模型
- 云上读写逻辑
- 智能体、一键主详、日志等业务能力本身

## 全局布局改造

### 1. 顶部 Header

Header 继续保留为全局外壳的一部分，但降级为“最小承载层”：

- 不再显示当前模块标题、副标题、解释文案
- 不再显示版本按钮、服务状态按钮、通知按钮、帮助按钮、账号按钮
- 高度显著缩减，只保留必要的留白、全局弹层挂载位置和少量全局状态容器
- 视觉上保持轻薄，避免形成第二导航条

实现上不应直接删除 Header 组件，而应改成最小模式，避免把以下现有能力一起打坏：

- 通知中心弹层定位
- 版本说明弹层定位
- 顶部 toast / 浮层安全区
- 后续其他模块可能仍依赖的统一外壳间距

### 2. 左侧导航底部用户区

左侧导航底部新增固定的“用户工作台入口”，成为全系统统一的全局入口承载位。

默认展示内容：

- 用户头像
- 用户名
- 身份标签

点击后弹出一个向上的玻璃风格面板，分为三组：

#### 状态组

- 当前版本号，例如 `V260407A`
- 服务状态，例如 `服务正常`

要求：

- 服务状态用一行状态项表现，不再单独做顶部按钮
- 版本项支持打开版本更新说明弹层

#### 全局操作组

- 通知中心
- 帮助 / 使用说明

要求：

- 仍保留现有通知已读状态与更新说明入口能力
- 不新增第二套通知逻辑，只迁移入口位置

#### 账户组

- 系统设置
- 账号管理
- 退出登录

要求：

- 管理员和普通员工共用同一个入口，但菜单项权限保持原有规则
- 返回路径清晰，不能出现“进入后出不来”或多层弹窗叠加

## 智能体中心改造

智能体中心是本次压缩的重点页面。

### 1. 模块首页头部

删除当前大面积模块标题区，不再在内容区顶部重复显示：

- 智能体中心标题
- 模块副标题
- 当前登录提示

改为一条紧凑工具栏，只保留真正影响操作的控件：

- 工厂 / 广场切换
- 智能体 / 知识库切换

要求：

- 不再出现大块空白横条
- 工具栏高度显著降低
- 工具栏应与内容区更贴合，而不是再做一层大容器

### 2. 工作室页头

工作室页面保留最小必要信息：

- 返回按钮
- 智能体头像
- 智能体名称
- 草稿版本名 / 版本号
- 训练 / 测试切换

但整体高度继续压薄：

- 减少上下 padding
- 减少大尺寸标题和说明文字
- 不再做大面积空白展示

### 3. 训练 / 测试内容区

训练页和测试页都以“消息区高度优先”为原则：

- 顶部状态条继续保留，但高度缩减
- token、清空对话等次级操作压缩成轻量控件
- 输入区工具能力与输入框进一步整合，避免上下分离过度
- 页面主滚动以消息区为主，而不是整个页面层层滚动

### 4. 输入区整合

训练页和测试页的输入区统一遵循：

- 模型选择、附件、联网、思考强度等能力贴近输入框布局
- 不再额外占据独立的大块区域
- 视觉上形成一个完整的输入容器，而不是多个上下切开的盒子

## 其他模块联动

本次不是只改智能体中心，而是全系统统一外壳。

需要一并联动检查的模块：

- 账号管理
- 系统设置
- 通知中心
- 版本更新说明
- 日志 / 用量统计类长列表页面
- 一键主详、视频、翻译、买家秀等仍依赖现有顶部留白的页面

联动目标：

- 去掉顶部模块标题后，不留下新的空白断层
- 内容区高度统一上移
- 列表、工作台、编辑页不出现新的截断和滚动异常

## 实施边界

### 包含

- `Header` 最小化
- `SidebarNavigation` 底部用户区聚合入口
- 通知 / 版本说明 / 帮助 / 账号入口迁移
- 智能体中心首页与工作室头部压缩
- 与壳层高度相关的内容区间距重算
- 相关 UI 架构测试更新

### 不包含

- 业务接口改造
- 智能体功能逻辑改造
- 日志统计口径变更
- 云端部署流程变更

## 风险与约束

### 风险 1：入口迁移导致功能“看起来消失”

对策：

- 左侧底部用户区必须始终可见
- 聚合面板分组清晰，避免把所有项堆成一列
- 版本和服务状态保留清晰文案，不只剩图标

### 风险 2：顶部缩减后弹层定位错位

对策：

- Header 保留最小壳层，不直接删掉组件
- 通知与更新说明优先复用现有浮层逻辑，只迁移触发入口

### 风险 3：页面高度收口不一致，出现截断或滚不动

对策：

- 统一检查主内容容器、模块容器、子页面容器的 `min-h-0 / overflow` 关系
- 保证“页面外壳可轻微回弹，核心内容区独立滚动”的现有交互原则不被破坏

## 验收标准

### 全局

- 顶部不再展示模块标题、副标题和右上角那排入口
- 左侧底部用户区可打开聚合面板
- 版本说明、服务状态、通知、帮助、系统设置、账号管理、退出登录都能从该入口使用

### 智能体中心

- 模块首页顶部高度明显缩小
- 工厂 / 广场 与 智能体 / 知识库 切换不再分散占位
- 工作室训练 / 测试页面首屏可见消息区明显增高
- 输入区工具和输入框形成更紧凑的一体化布局

### 回归

- 账号管理、系统设置能正常进入和返回
- 通知中心和版本说明弹层能正常打开
- 其他模块不存在新的顶部空白、内容截断、滚动异常

## 实施顺序

1. 先改全局壳层：Header 最小化，SidebarNavigation 底部用户区聚合
2. 再改智能体中心：首页头部、工作室页头、训练/测试区压薄
3. 再扫其他模块页面：修正顶部留白和内容区高度
4. 最后做 UI 架构测试和本地交互回归
