# 智能体聊天工作台重构设计

## 目标

将当前过于简陋的智能体使用页重构为独立聊天工作台，显著提升视觉层级、信息组织和使用效率，同时为后续接入更多模型 API 留出稳定扩展位。

本次重构聚焦使用端，不改坏已有管理端能力，并保持本地模式与云上模式都可运行。

## 已确认决策

- 使用端采用独立“聊天工作台”体验，不再停留在简单页签式聊天区
- 整体布局采用 `A` 方案：左侧双层导航 + 右侧沉浸式聊天区
- 会话列表默认按智能体分组
- 普通用户只能在智能体预设允许的模型范围内切换
- 附件能力按模型真实能力动态开放，图片和文档都支持，只要模型支持
- 用户头像需要纳入账号管理，支持默认头像与上传头像
- 会话支持删除，并继续保持严格的用户隔离
- 后续会接入更多外部 API，因此模型能力判断必须由服务端配置驱动，不能写死在前端

## 页面结构

### 1. 总体布局

聊天工作台采用三栏布局：

- 左栏一：智能体列表
- 左栏二：按智能体分组的会话列表
- 右侧主区：聊天区、顶部控制区、底部输入区

布局目标：

- 左侧解决“找谁聊”和“找哪段会话”
- 右侧只做当前会话上下文与输入输出
- 聊天区成为视觉焦点，避免当前页面的低密度卡片堆叠

### 2. 智能体列表

展示已发布、当前用户可使用的智能体：

- 智能体图标
- 智能体名称
- 简短说明
- 当前是否选中

交互：

- 点击切换当前智能体
- 切换后右侧会话列表只显示该智能体分组
- 若该智能体无会话，展示空状态并引导新建

### 3. 会话列表

会话按智能体分组展示，每组显示：

- 智能体图标和名称
- 该智能体下的会话条目

每条会话显示：

- 会话标题
- 最近更新时间
- 当前使用模型摘要
- 删除入口

交互：

- 支持搜索会话标题
- 支持新建会话
- 支持删除会话，并二次确认
- 删除只删除当前用户自己的会话、消息和附件引用

### 4. 聊天主区

消息布局规范：

- 智能体消息在左，头像为智能体图标
- 用户消息在右，头像为当前用户头像
- 支持 Markdown 文本展示
- 支持消息发送中、失败重试、空状态

第一阶段不做：

- 消息编辑
- 消息分支
- 多轮草稿
- 消息级 pin / 收藏

### 5. 顶部控制区

顶部控制区包含当前会话相关控制项：

- 当前智能体名称
- 当前模型
- 思考强度
- 联网开关
- 附件能力提示

这些控制项必须按模型能力动态显示：

- 不支持思考强度的模型，不显示该控件
- 不支持联网的模型，不显示联网开关
- 不支持图片或文件上传的模型，对应上传入口隐藏或禁用并解释原因

### 6. 输入区

输入区包括：

- 多行文本输入框
- 图片上传入口
- 文件上传入口
- 已选择附件预览区
- 发送按钮

行为规则：

- 如果当前模型支持图片但不支持文件，只显示图片上传
- 如果当前模型支持文件但不支持图片，只显示文件上传
- 如果均支持，则同时显示
- 如果都不支持，只保留纯文本输入

## 视觉风格

整体风格延续当前管理端的浅色玻璃态方向，但更偏“成熟工作台”：

- 大面积留白，降低压迫感
- 聊天主区层级最强
- 左侧列表采用轻玻璃和轻描边
- 顶部控制区像工具条，不做传统表单堆叠
- 消息区保持阅读性，避免装饰过度

视觉目标：

- 比当前版本更成熟
- 不做花哨的概念稿
- 真实可长期使用

## 数据模型变更

### 1. 用户头像

为用户新增：

- `avatarUrl`
- `avatarPreset`

要求：

- 本地 JSON store 与 MySQL schema 均兼容
- 若未设置头像，使用默认预设

### 2. 智能体版本模型限制

为智能体版本新增：

- `allowedChatModels`
- `defaultChatModel`

意义：

- 每个智能体可限制普通用户可选模型范围
- 普通用户不能任意切换到平台所有模型

### 3. 模型能力目录

系统配置中的模型目录扩展为能力驱动结构。每个聊天模型至少包含：

- `id`
- `label`
- `supportsImageInput`
- `supportsFileInput`
- `supportsWebSearch`
- `supportsReasoningLevel`
- `reasoningLevels`

后续接入新 API 时，只需扩展服务端能力表，无需重构聊天前端。

### 4. 会话配置快照

聊天会话增加：

- `selectedModel`
- `reasoningLevel`
- `webSearchEnabled`

要求：

- 会话切回时可恢复之前的会话配置
- 若模型能力变化导致旧配置失效，服务端需回退到安全默认值

### 5. 附件元数据

聊天消息或会话附件关联增加：

- 文件名
- 文件类型
- 资源 URL / assetId
- 上传用户
- 所属会话

要求：

- 继续按用户和会话强隔离
- 不允许跨用户读取附件元数据

## API 变更

### 新增或扩展的使用端接口

- `GET /api/chat/agents`
  - 返回可用智能体及其默认模型、允许模型、图标信息
- `GET /api/chat/sessions`
  - 返回当前用户会话，支持按智能体过滤
- `POST /api/chat/sessions`
  - 创建会话时允许传入初始模型配置
- `DELETE /api/chat/sessions/:id`
  - 删除当前用户自己的会话
- `GET /api/chat/sessions/:id/messages`
  - 返回消息与附件元数据
- `POST /api/chat/sessions/:id/messages`
  - 支持文本 + 附件 + 当前会话选项
- `PATCH /api/chat/sessions/:id`
  - 更新会话的当前模型、思考强度、联网状态

### 账号资料相关接口

新增“当前用户资料更新”能力，允许用户本人修改：

- 显示名
- 头像上传
- 默认头像预设

总管理员原有用户管理能力保留，不混淆“管理别人”与“修改自己资料”。

## 能力判断规则

### 1. 模型可选范围

最终可选模型集合 = `智能体允许模型集合` 与 `系统当前可用聊天模型集合` 的交集。

若交集为空：

- 前端显示异常状态
- 禁止发消息
- 引导管理员修正智能体配置

### 2. 思考强度

只有模型能力标记 `supportsReasoningLevel = true` 时显示。

默认档位：

- 若会话已有配置，优先会话配置
- 否则使用模型默认档位
- 无默认档位时使用第一档

### 3. 联网

只有模型标记支持联网时显示。

若用户打开联网但当前模型不支持：

- 前端应立即关闭该选项
- 服务端仍再次校验，防止脏请求

### 4. 附件

上传入口依据模型能力分别开放：

- 支持图片：开放图片上传
- 支持文件：开放文档上传

服务端必须再次校验能力，不允许仅依赖前端判断。

## 权限与隔离

### 1. 使用权限

- 仅内部登录用户可进入聊天工作台
- 仅已发布智能体可被普通用户使用

### 2. 数据隔离

会话、消息、附件全部按当前用户过滤：

- 不能读取别人的会话
- 不能读取别人的附件
- 删除操作只能删除自己的数据

### 3. 管理权限

智能体允许模型集合由管理端配置，普通用户不可修改。

## 第一阶段实施范围

第一阶段要完成：

- 独立聊天工作台 UI
- 左侧智能体列表
- 按智能体分组的会话列表
- 新建会话、删除会话
- 智能体/用户双头像消息布局
- 用户头像设置
- 基于模型能力的顶部控制区
- 图片/文件上传入口与服务端校验
- 会话选项持久化

第一阶段暂不做：

- 外部插件系统
- 自定义工具调用
- 多智能体协同会话
- 消息编辑与分支
- 高级搜索与全局知识联查

## 实施顺序

### 阶段 1：数据与能力模型

- 扩展用户头像字段
- 扩展智能体版本模型集合字段
- 扩展模型能力目录
- 扩展聊天会话配置字段

### 阶段 2：聊天接口补齐

- 新增会话删除
- 新增会话配置更新
- 扩展发消息接口，支持附件和选项
- 服务端能力校验

### 阶段 3：使用端 UI 重构

- 重写聊天工作台布局
- 接入头像、分组会话、控制区、输入区
- 打通空状态、错误态、加载态

### 阶段 4：账号资料完善

- 账号管理增加个人头像设置
- 支持默认头像与上传头像

### 阶段 5：联调与验证

- 本地模式验证
- MySQL 模式兼容验证
- 权限与隔离验证
- 构建与测试验证

## 风险与处理

### 1. 模型能力不一致

不同 API 的能力差异会越来越大，必须坚持“服务端能力定义，前端只消费”的原则，避免前端硬编码。

### 2. 附件接入半成品风险

如果某个模型链路尚未真正支持文件问答，前端不能只放按钮。必须由服务端能力配置决定是否真正开放。

### 3. 会话配置脏数据

历史会话中的模型或选项可能失效。服务端恢复会话时要进行规范化回退，保证页面能打开。

## 验收标准

- 使用端不再是粗糙三栏卡片布局，而是可长期使用的聊天工作台
- 用户能清晰区分当前智能体、当前会话和当前模型
- 会话按智能体分组，支持删除
- 用户头像与智能体头像均正确展示
- 模型能力控制项按服务端配置动态显示
- 图片和文件上传按模型能力开放
- 所有会话、消息、附件继续严格按用户隔离
- 本地与云上均兼容，不破坏已有数据
