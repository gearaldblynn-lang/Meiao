# 一键主详统一预设库管理中心设计

- 日期：2026-04-29
- 范围：`首图 / 主图 / 详情 / SKU`
- 目标：将现有割裂的文字预设和图片预设整合为统一的图文预设库，并提供可长期使用的卡片式管理中心。

## 1. 背景

当前一键主详的预设库存在三个结构性问题：

1. `主图 / 详情` 仅保存文字分析结论，不保存对应参考图，复用时不直观。
2. `首图 / SKU` 仅保存单图，不保存分析摘要、维度等上下文，复用信息不完整。
3. 预设库入口是轻量弹窗，只有查看/应用/删除的最小能力，不具备成熟的新增、编辑、分类管理能力。

这导致用户虽然可以临时存预设，但无法形成长期可维护、可复用、可筛选的设计资产库。

## 2. 设计目标

本次改造必须满足以下目标：

1. 四个子功能统一进入同一套预设管理中心。
2. 所有预设都必须支持图文一体化存储。
3. 主视图采用卡片库，优先保证看图直观。
4. 预设必须支持新增、编辑、删除、应用。
5. 老数据必须自动迁移，不能丢失已有预设。
6. 应用预设时必须能把完整参考上下文正确回填回当前子功能。

## 3. 非目标

本次不做以下内容：

1. 不引入服务端预设管理接口。
2. 不做跨账号共享、团队协作、权限控制。
3. 不做批量删除、批量标签编辑等高级后台能力。
4. 不做超出一键主详模块的通用素材库系统。

## 4. 信息架构

### 4.1 管理中心入口

将现有轻量弹窗升级为统一的“预设库管理中心”。

入口仍然从四个子功能内部进入，但进入后看到的是同一套管理界面，而不是子功能各自维护一套分裂弹窗。

### 4.2 主界面结构

页面结构分为四部分：

1. 顶部分类 Tab：`首图 / 主图 / 详情 / SKU`
2. 工具栏：搜索、筛选、排序、`新增预设`
3. 主内容区：卡片网格
4. 详情区域：抽屉或大弹层，用于查看、编辑、删除、应用

### 4.3 卡片视图

每张卡片展示以下信息：

1. 封面图
2. 预设名称
3. 预设所属子功能标签
4. 摘要前 2-3 行
5. 参考维度标签
6. 更新时间
7. 快捷操作：`应用 / 编辑 / 删除`

设计原则：

1. 首要信息是图，不是文字。
2. 文字只承担识别和筛选辅助，不抢视觉主位。
3. 任何预设都必须能通过封面快速判断是否值得点开。

## 5. 统一数据模型

## 5.1 新模型

废弃当前割裂的三套集合：

- `textPresets`
- `firstImageImagePresets`
- `skuImagePresets`

统一为单一集合：

```ts
interface OneClickReferencePreset {
  id: string;
  name: string;
  subMode: 'first_image' | 'main_image' | 'detail_page' | 'sku';
  coverImageUrl: string;
  referenceImageUrls: string[];
  summary: string;
  detail: string;
  referenceDimensions: OneClickReferenceDimension[];
  tags: string[];
  createdAt: number;
  updatedAt: number;
}
```

```ts
interface OneClickReferencePresetLibrary {
  presets: OneClickReferencePreset[];
}
```

### 5.2 字段约束

1. `name` 必填。
2. `subMode` 必填，且只能属于四个子功能之一。
3. `coverImageUrl` 必填。
4. `referenceImageUrls` 至少 1 张。
5. `summary` 必填，面向卡片列表展示。
6. `detail` 可与 `summary` 相同，但不能为空字符串。
7. `referenceDimensions` 允许为空数组，但从分析结果保存时应尽量带上。
8. `tags` 为可选补充标签，不参与核心业务。

### 5.3 为什么要统一

统一模型的目的不是抽象好看，而是解决三个实际问题：

1. 同一套页面可以稳定渲染四类预设。
2. 保存、编辑、删除、应用都走同一条行为链路，减少重复代码和漂移。
3. 后续增加筛选、排序、搜索时不需要分别兼容三套旧结构。

## 6. 预设来源与保存规则

### 6.1 从当前分析结果保存

#### 首图

保存时自动带入：

1. 当前参考图
2. 封面图
3. 当前分析摘要（如果有）
4. 当前参考维度
5. 子功能标识 `first_image`

#### 主图 / 详情

保存时自动带入：

1. 当前设计参考图列表中的首张图作为封面图
2. 全部参考图列表
3. 当前参考分析摘要
4. 当前参考分析完整内容
5. 当前参考维度
6. 对应 `subMode`

这里的关键要求是：`主图 / 详情` 不能再只存文字，必须把参考图一起存进去。

#### SKU

保存时自动带入：

1. 当前 style reference 图
2. 封面图
3. 当前说明性摘要（如无独立分析结果，可用用户填写说明或默认摘要）
4. 对应 `subMode`

### 6.2 手动新增预设

用户可在预设库内直接新建：

1. 上传 1 张或多张参考图
2. 指定封面图
3. 选择适用子功能
4. 填写名称
5. 填写摘要
6. 填写完整说明
7. 选择参考维度
8. 填写标签

### 6.3 编辑预设

用户可编辑：

1. 名称
2. 子功能
3. 封面图
4. 参考图列表
5. 摘要
6. 完整说明
7. 参考维度
8. 标签

不可编辑：

1. `id`
2. `createdAt`

编辑后必须更新 `updatedAt`。

### 6.4 删除预设

删除必须二次确认。

删除只影响预设库本身，不影响当前工作区已经应用的内容。

## 7. 预设应用规则

### 7.1 首图

应用预设后回填：

1. `designReferences`
2. `uploadedDesignReferenceUrls`
3. `lastStyleUrl`
4. 如果预设带有说明文字，则同步到参考分析区域

### 7.2 主图 / 详情

应用预设后回填：

1. `designReferences`
2. `uploadedDesignReferenceUrls`
3. `lastStyleUrl`
4. `referenceAnalysis.summary`
5. `referenceDimensions`

如果预设 `detail` 与 `summary` 不同，详情区域优先显示 `detail`，分析摘要区域使用 `summary`。

### 7.3 SKU

应用预设后回填：

1. style reference 图
2. 相关说明文本
3. 参考维度（若 SKU 后续开始使用）

### 7.4 应用原则

1. 应用预设只替换参考上下文，不直接修改用户已经产出的策划方案。
2. 若当前工作区已有参考图，应用前必须二次确认是否覆盖。
3. 应用成功后必须有明确 toast 提示。

## 8. 页面交互设计

### 8.1 列表区

支持：

1. 分类 Tab 切换
2. 搜索名称/摘要
3. 按更新时间排序
4. 按维度筛选
5. 按是否含多图筛选

### 8.2 卡片交互

卡片支持：

1. 点击打开详情
2. 悬浮展示快捷操作
3. 当前选中态高亮

### 8.3 详情面板

详情面板必须展示：

1. 封面大图
2. 参考图缩略图列表
3. 预设名称
4. 子功能标签
5. 摘要
6. 完整说明
7. 参考维度
8. 更新时间
9. 操作按钮：`应用 / 编辑 / 删除`

### 8.4 新增 / 编辑表单

必须废弃 `window.prompt`，改为正式表单。

表单字段：

1. 名称
2. 子功能
3. 封面图
4. 参考图列表
5. 摘要
6. 完整说明
7. 参考维度
8. 标签

校验规则：

1. 名称不能为空
2. 至少有 1 张参考图
3. 摘要不能为空
4. 封面图必须属于参考图集合中的一张，或由参考图默认推导

## 9. 迁移方案

## 9.1 迁移目标

旧状态必须兼容：

1. `textPresets`
2. `firstImageImagePresets`
3. `skuImagePresets`

### 9.2 迁移规则

#### textPresets -> presets

映射为：

1. `subMode = sourceSubMode`
2. `summary = summary`
3. `detail = summary`
4. `referenceDimensions = referenceDimensions`
5. `referenceImageUrls = []` 时需要补兜底

由于旧文字预设不带图，迁移时必须处理这个缺口：

1. 若旧数据本身没有图，则允许生成一个“无图占位”状态，但该预设在卡片上显示缺图标识。
2. 更推荐在迁移时，如果本地还能找到关联参考图则优先补入；如果没有，就保留为无图预设并允许用户后续编辑补图。

为了满足新模型“至少 1 张图”的约束，迁移层需要单独放宽：历史数据可短期保留无图状态，但新建/编辑保存时必须补齐。

#### firstImageImagePresets -> presets

映射为：

1. `subMode = first_image`
2. `coverImageUrl = imageUrl`
3. `referenceImageUrls = [imageUrl]`
4. `summary = name`
5. `detail = name`

#### skuImagePresets -> presets

映射方式同上，`subMode = sku`。

### 9.3 兼容期策略

1. 读取时兼容旧结构并自动标准化到新结构。
2. 一旦保存回本地状态，统一写回新结构。
3. 旧字段不再继续作为主写入目标。

## 10. 组件拆分建议

建议新增或重构以下组件：

1. `ReferencePresetManager.tsx`
   - 统一管理中心容器
2. `ReferencePresetCard.tsx`
   - 卡片项
3. `ReferencePresetDetailPanel.tsx`
   - 详情抽屉/弹层
4. `ReferencePresetEditorModal.tsx`
   - 新增/编辑表单
5. `referencePresetUtils.mjs`
   - 数据标准化、迁移、搜索、筛选、排序、应用映射

现有 `ReferencePresetLibraryModal.tsx` 不再继续扩展为主入口，可保留为过渡层或直接由新组件替代。

## 11. 状态与存储

继续沿用当前一键主详本地持久化方式，不增加新后端接口。

需要改动：

1. `types.ts`：替换旧预设类型为新统一类型
2. `utils/appState.ts`：补充旧数据迁移和新结构默认值
3. `OneClickModule.tsx`：统一保存、删除、编辑、应用入口
4. `ConfigSidebar.tsx / SkuSidebar.tsx`：入口改为打开管理中心
5. 四个子模块：接入统一应用逻辑

## 12. 测试要求

至少补以下测试：

1. 旧结构迁移到新结构不丢数据
2. 主图/详情保存预设时会连同参考图一起保存
3. 首图/SKU 保存后生成完整统一预设对象
4. 编辑预设后 `updatedAt` 更新且内容持久化
5. 删除预设后状态正确移除
6. 应用预设时能正确回填到四个子功能的参考状态
7. 无图历史预设在新卡片库中能正常显示缺图状态，不会导致崩溃

## 13. 风险与处理

### 风险 1：历史文字预设没有图

处理：

1. 迁移时允许历史数据短期处于缺图状态
2. UI 明确提示该预设建议补图
3. 用户编辑保存后必须补齐图片

### 风险 2：四个子功能应用字段不一致

处理：

1. 统一封装“预设 -> 子功能状态”的映射函数
2. 避免在四个组件里各写一套零散回填逻辑

### 风险 3：继续在旧弹窗上叠补丁

处理：

1. 不再继续给旧弹窗堆功能
2. 以统一管理中心替代旧弹窗主入口

## 14. 验收标准

以下条件全部满足才算完成：

1. 四个子功能都能进入统一预设库管理中心
2. 主视图为卡片库，且看图优先
3. 主图/详情预设可同时看到参考图和分析结果
4. 四类预设都支持新增、编辑、删除、应用
5. 旧数据升级后不丢失
6. 应用预设后能正确回填当前子功能
7. 不再使用 `window.prompt` 作为预设新增或编辑入口
