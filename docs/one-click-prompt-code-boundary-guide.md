# 一键主详 Prompt / 代码规则 / 边界说明

更新日期：2026-04-30

本文档用于统一一键主详链路里的三类规则，避免后续把本该写在 Prompt 里的要求、写到代码里，或者把本该由代码保证的行为、错误地塞进 Prompt。

## 1. 三层规则定义

### 1.1 Prompt 规则

定义：直接发给模型的规则，用来约束模型如何理解任务、如何输出内容、如何组织结构。

典型特征：
- 模型必须自己“理解并执行”
- 与输出内容、表达方式、格式结构直接相关
- 不依赖前端状态或交互逻辑

例如：
- 必须复刻参考图风格、版式、信息层级
- 文案内容排版必须按固定格式输出
- 只有中文引号内的文字是最终渲染正文
- 必须去除参考图中的 logo、品牌名、店铺名
- 不得虚构卖点、商品形态、促销信息

### 1.2 代码规则

定义：程序提前保证的行为规则，用来约束链路、状态、数据、交互和任务执行方式。

典型特征：
- 不依赖模型自己理解
- 即使模型出错，程序也要兜底
- 属于工作台、状态管理、任务编排、持久化、并发控制

例如：
- 编辑后的策划内容必须覆盖原方案再送去生图
- 多张参考图的任务允许并行，不应串行阻塞
- 工作台必须保留多个项目，不能被新任务顶掉
- 裂变必须新开项目，而不是污染旧项目
- 生图 prompt 自动拼接统一护栏，不把策划示例再次塞给生图模型
- 页面刷新后项目列表要从持久化状态恢复

### 1.3 边界说明

定义：明确一条要求到底由 Prompt 保证、由代码保证，还是双保险但主责归其中一方。

原则：
- 输出内容是否正确，优先看 Prompt
- 链路是否稳定、状态是否一致，优先看代码
- 同一件事只允许一个主责层，另一层最多做辅助兜底

## 2. 当前一键主详的规则分层

### 2.1 策划模型 Prompt 负责什么

文件入口：
- `src/services/arkService.ts`
- `generateMarketingSchemes`
- `generateFirstImageReplicationSchemes`
- `generateSkuSchemes`

策划 Prompt 主责：
- 使用 RTCFE 结构组织任务
- 约束输出字段、字段顺序、解析锚点
- 约束策划内容的完整性、准确性、风格一致性
- 约束文案内容排版的固定输出结构
- 约束不得自由发挥、不得输出旧格式、不得编造
- 约束首图复刻时必须基于参考图做商品和卖点迁移

不应交给策划 Prompt 负责的内容：
- 前端显示几个方案
- 多图任务是否并行
- 编辑后的内容是否真的被写回状态
- 裂变是否新开项目
- 是否保存预设、保存几张图
- 工作台是否保留历史项目

### 2.2 生图模型 Prompt 负责什么

文件入口：
- `src/modules/OneClick/FirstImageSubModule.tsx`
- `src/modules/OneClick/MainImageSubModule.tsx`
- `src/modules/OneClick/DetailPageSubModule.tsx`
- `src/modules/OneClick/SkuSubModule.tsx`
- `src/modules/OneClick/generationPromptUtils.ts`

生图 Prompt 主责：
- 严格保持产品一致性
- 严格复刻参考图风格、版式、信息层级、设计细节
- 严格按策划文案内容排版执行
- 严格去除参考图与素材中的竞品 logo / 他牌标识
- 严格限制最终渲染文案语言
- 严格区分“括号内排版要求”和“引号内正文文案”

不应交给生图 Prompt 负责的内容：
- 生图前是否自动裁剪掉策划示例
- 哪张图是当前活动项目
- 是否使用编辑后的最新方案
- 是否创建新项目
- 项目切换、项目删除、项目恢复

### 2.3 代码层主责什么

文件入口：
- `src/modules/OneClick/OneClickModule.tsx`
- `src/utils/appState.ts`
- 四个子模块文件

代码规则主责：
- 子模块状态管理
- 当前工作台与历史项目的并存
- 新任务前自动新建项目，避免覆盖旧项目
- 项目切换、删除、持久化恢复
- 裂变新建项目
- 编辑内容写回当前状态，并被后续生图读取
- 上传素材与参考图的角色区分
- 多参考图任务编排方式
- Prompt 组装时追加统一护栏、去掉无关示例

## 5. 生图 Prompt 的推荐结构

生图 Prompt 只保留三段：

### 5.1 前缀：硬约束

只写以下几类：
- 产品一致性
- 参考图复刻要求
- 去 logo / 去品牌 / 去原文案
- 禁止虚构
- 文案语言与渲染硬约束

要求：
- 同一件事只说一次
- 用执行命令，不用解释型废话
- 尽量中文

### 5.2 主体：当前策划结果

只放：
- 设计意图
- 画面描述
- 文案内容排版

不放：
- 输出规范示例
- RTCFE 解释
- 给策划模型看的 few-shot 样例

### 5.3 后缀：统一护栏

由代码统一追加的内容：
- 目标语言
- 引号内正文才渲染
- 括号内是排版指令
- 字段名、冒号、说明文字不得出现在画面中

## 6. 策划 Prompt 的推荐结构

策划 Prompt 必须按 RTCFE 写，但要注意它和生图 Prompt 不是同一个东西。

策划 Prompt 应包含：
- R：角色
- T：任务
- C：约束
- F：格式
- E：示例

策划 Prompt 应强调：
- 输出字段固定
- 输出顺序固定
- 文案内容排版固定
- 示例只给策划模型看

策划 Prompt 不应承担：
- 项目管理
- 状态保存
- 并发执行
- 页面交互

---

## 7. 后续修改时的检查顺序

以后每次改一键主详链路，按这个顺序检查：

1. 先判断问题属于哪一层
   - 是模型输出偏了，还是程序链路错了
2. 如果是内容、格式、渲染要求问题
   - 先看 Prompt
3. 如果是覆盖旧项目、编辑不生效、并发不对、保存重复、状态丢失
   - 先看代码
4. 如果两边都能做
   - 必须明确主责层，只留一层主责，另一层仅做辅助兜底

## 8. 当前一键主详的长期规范

### 8.1 Prompt 层长期规范

- 所有策划 Prompt 必须使用 RTCFE 结构
- 所有生图 Prompt 必须使用“硬约束 + 当前策划结果 + 统一护栏”三段式
- 生图 Prompt 不得再带策划示例
- 生图 Prompt 中相同含义的约束不得重复堆叠
- 文案内容排版相关规则只保留一套标准表述

### 8.2 代码层长期规范

- 所有一键主详子模块都必须支持多项目工作台
- 任何重新策划、重新裂变都不能覆盖旧项目
- 编辑后的方案必须是唯一真实来源，后续生图只能读取当前最新编辑值
- 参考图、产品图、logo 图、历史结果图必须始终区分角色
- 持久化恢复必须兼容旧状态与新状态

### 8.3 边界层长期规范

- 不要用 Prompt 去补代码状态问题
- 不要用代码去替代模型内容理解问题
- 不要把同一条约束在策划 Prompt、生图 Prompt、代码注入后缀里各写一遍
- 每条规则都要有主责归属

---

## 9. 直接结论

以后改一键主详，先问自己三句话：

1. 这条要求是让模型理解，还是让程序保证？
2. 如果模型不可靠，程序是否必须兜底？
3. 这条要求现在是不是已经在别的层写过一次了？

只要这三句先过一遍，后面 Prompt 和代码就不会继续互相污染。
